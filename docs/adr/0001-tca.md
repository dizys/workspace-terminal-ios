# ADR-0001: The Composable Architecture for state management

- **Status:** Accepted
- **Date:** 2026-04-17

## Context

The app's core loop is async streams (PTY bytes) feeding into UI state, plus reducer-style transitions for connection state, multi-tab management, and lifecycle actions. State coherence matters: a connection-state regression (e.g., showing "Connected" while transport is reconnecting) is a user-visible bug.

Options considered:

1. **Vanilla SwiftUI + `@Observable`** — minimal ceremony, but state spreads across many view models, integration with long-lived async effects becomes ad hoc, and testability of cross-screen flows is poor.
2. **MVVM with Combine** — proven, but Combine is in soft maintenance mode and Apple's direction is async/await + `Observation`.
3. **The Composable Architecture (TCA)** — opinionated reducer architecture, first-class support for async effects, excellent testability via `TestStore`, dependency injection via `@Dependency`.

## Decision

Use TCA from day 1.

## Consequences

**Positive:**
- Reducers are pure → trivial to unit-test every state transition.
- Effects are first-class and `.cancellable(id:)` — clean lifecycle for long-lived PTY connections.
- `@Dependency` makes injecting fakes (CoderAPIClient, PTYClient, Clock, Keychain) frictionless.
- Time-travel debugging during development.

**Negative:**
- Learning curve for contributors (closed-source for now, but a future hire/contractor would face it).
- More boilerplate than vanilla `@Observable`.
- Couples us to Point-Free's release cadence; mitigated by TCA being battle-tested and stable.

**Mitigations:**
- One ADR per non-trivial reducer pattern as we encounter them.
- Keep features small; avoid "god reducers".
