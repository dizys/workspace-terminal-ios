# ADR-0006: OIDC required at v1

- **Status:** Accepted
- **Date:** 2026-04-17

## Context

Coder deployments authenticate users via:

1. Username/password (built-in)
2. GitHub OAuth
3. OIDC (Okta, Auth0, Keycloak, Azure AD, Google Workspace, etc.)

Many enterprise self-hosted deployments **disable** username/password and require OIDC. Without OIDC support, the app is unusable for them.

## Decision

**OIDC support is required at v1.** Login screen probes `/api/v2/users/authmethods` and renders only the available methods.

## Consequences

**Positive:**
- App works for enterprise deployments out of the box.
- Larger addressable market.
- Validates the harder auth flow early; password is trivial by comparison.

**Negative:**
- More implementation effort in M1 (~1.5 wk vs ~1 wk).
- `ASWebAuthenticationSession` UX has rough edges (custom CA limitations on iOS — see [coder-integration.md](../coder-integration.md#custom-ca-caveat)).
- PKCE + custom URL scheme handling adds attack surface to validate.

**Mitigations:**
- Document the custom-CA-on-iOS limitation prominently in support docs.
- Use first-party `ASWebAuthenticationSession` (not a third-party OIDC lib) to minimize attack surface.
- Comprehensive unit tests on OIDC callback parsing — easy to get wrong, security-sensitive.
